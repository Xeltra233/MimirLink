# 角色卡与预设编写工作流

本文只描述通用写法，不绑定任何具体角色。目标是把“角色素材”和“运行方式”拆开，避免所有东西都塞进一段系统提示里，导致角色死板、上下文混乱、正文和思考泄漏。

## 核心分工

角色卡负责“这个人是谁”。

预设负责“模型这次怎么工作”。

世界书负责“在什么世界、有哪些长期事实”。

正则/后处理负责“哪些内容显示，哪些内容隐藏或转换”。

前端/靶场负责“能不能看清每一层到底发生了什么”。

不要让角色卡承担预设的职责。角色卡里可以写性格、经历、关系、禁忌、说话习惯，但不应该塞一整套 CoT 流程、输出协议、工具调用协议、上下文压缩策略。反过来，预设也不应该写死某个角色的具体经历和台词，否则迁移到别的角色会污染。

## 编写顺序

1. 先写角色卡骨架。

   最小结构包括：身份、性格核心、行为边界、关系处理、语言风格、典型反应、禁忌。每一项都写成可执行的判断，而不是形容词堆叠。

   差写法：

   ```text
   角色很聪明、很温柔、很有魅力。
   ```

   好写法：

   ```text
   遇到低信息问候时先简短接住，不主动展开长篇解释。
   对熟人可以轻微调侃，对陌生人先保持距离。
   用户表达疲惫时先承认状态，再给一句可接的话。
   ```

2. 再写世界书。

   世界书只放可复用事实：地点、组织、事件、术语、长期关系、世界规则。不要把每轮回复策略塞进世界书。世界书条目要有触发词和边界，避免只要提到一个词就把大段设定灌进 prompt。

3. 再写预设。

   预设定义运行结构：消息解码、上下文优先级、思考者身份、正文标签、字数控制、低信息处理、工具使用、输出后检查。它应该尽量角色无关，能套到不同角色上。

4. 最后写正则和显示规则。

   如果预设要求输出 `<thinking>` 和 `<content>`，就必须有明确的正文提取规则。用户可见内容只取 `<content>`；`<thinking>` 只用于调试或隐藏展示。不要依赖模型“自觉不泄漏”。

## 角色卡怎么写

角色卡要给模型“材料”，不要给模型“流程”。推荐分块：

```text
【身份】
角色是谁，当前处境是什么，和用户/群聊/场景是什么关系。

【性格核心】
3-5 条最稳定的性格驱动。每条必须能影响具体回复。

【行为边界】
什么情况下认真，什么情况下调侃，什么情况下收住。

【语言风格】
句长、语气、是否用动作、是否用口头禅、是否允许反问。

【关系规则】
对陌生人、熟人、敌对者、亲近者分别怎么处理。

【典型反应】
给 5-10 个场景反应，不要只给漂亮台词。

【禁忌】
明确哪些行为会破坏角色，例如过度解释、复读口癖、主动把话题拐到旧梗。
```

角色卡里的示例要少而准。示例太多会让模型复读示例句式，尤其是固定反问、固定结尾、固定梗。示例应该覆盖“如何判断”，而不是只展示“应该说什么”。

## 角色卡规范化

角色卡也需要规范化。一个成熟角色卡不是“人设长文”，而是由静态素材、动态状态、触发条目、显示/隐藏规则共同构成。

推荐结构：

```text
card-meta              卡片元信息：名称、版本、作者备注、适配预设、依赖脚本/正则/世界书。
identity               身份：角色是谁，当前处境，和用户/场景的基本关系。
personality-palette    性格调色盘：底色、主色调、点缀、衍生行为。
behavior-rules         行为规则：遇到不同场景时怎么行动和说话。
relationship-rules     关系规则：陌生、熟悉、亲近、敌对、特殊关系如何变化。
speech-style           语言风格：句长、语气、口癖上限、动作描写比例。
scenario               当前场景：只写开局状态，不写死后续剧情。
first-message          开场白：给用户入口，不要承担全部设定说明。
examples               示例对话：少量、高质量、覆盖判断，不奖励固定句式。
character-book         内嵌世界书：长期事实、术语、状态条目、阶段条目。
variables              状态变量：好感、信任、关系阶段、任务进度等。
display-rules          显示规则：状态栏、手机、小窗、变量更新如何展示或隐藏。
compatibility          兼容说明：需要哪些预设、脚本、正则、模型能力。
```

### 性格调色盘

写卡预设里常见“调色盘”方法，它比单纯写性格标签更稳定。推荐写法：

```text
底色：角色最底层的稳定驱动力。
主色调：日常最常出现的外显气质。
点缀：偶尔出现、制造反差的性格边缘。
衍生：每个性格在具体场景中的行为表现。
二次解释：作者对这些衍生为什么成立的解释。
```

不要只写“温柔、聪明、冷淡、疯批”。每个词都要展开成行为：面对陌生人怎样、被误解怎样、被关心怎样、压力下怎样、关系变化后怎样。

### 动态阶段

如果角色会随关系或剧情变化，就不要把所有阶段混在一段描述里。应该拆成阶段：

```text
阶段变量：用什么变量判断阶段，例如好感、信任、关系状态、剧情节点。
阶段条件：每个阶段的边界不能重叠，也不能遗漏。
阶段表现：该阶段独有的语言、行为、心理防线、关系处理。
通用表现：不随阶段变化的底层性格。
阶段解释：为什么这个阶段会这样表现。
```

动态阶段适合放在世界书常驻条目或 EJS/MVU 条目里，不要把所有阶段同时塞进角色描述。否则模型会同时拿到互相冲突的人设。

### 角色卡依赖声明

角色卡如果依赖预设、世界书、正则、脚本、小窗 UI、变量系统，就必须在作者备注或 meta 中写清楚：

```text
适配预设：...
必需世界书：...
必需正则：...
必需脚本：...
可选模块：...
不支持/未测试：...
导入顺序：先导入角色卡，再导入世界书/正则/脚本，或反过来。
```

否则用户只导入 PNG/JSON 角色卡时，状态栏、变量更新、隐藏思维链、手机 UI 可能全部失效。

### 角色卡输出 Schema

写卡预设用于“生产角色卡”时，必须要求结构化输出。推荐输出 schema：

```yaml
name: 角色名
description: |
  身份、外观、经历、当前状态
personality: |
  性格调色盘、行为衍生、关系处理
scenario: |
  当前开局场景和限制
first_mes: |
  开场白
mes_example: |
  示例对话
creator_notes: |
  使用说明、依赖、禁忌、适配预设
character_book:
  entries:
    - comment: 条目名
      keys: [触发词]
      content: 条目内容
      strategy: constant/selective
      position: before/after_character_definition
variables:
  stat_data:
    角色名:
      好感度: 0
      关系状态: 陌生
```

写卡输出不要只给自然语言大纲。大纲还需要二次手工整理，容易漏字段。结构化输出更方便导入、审查和自动转换。

### 反八股检查

写卡时要检查“八股化”：

- 只写形容词，不写行为。
- 只写极端标签，不写日常表现。
- 大量模糊词和廉价比喻。
- 每个角色都用同一套微表情、同一套语气描写。
- 示例对话像作者解释设定，而不是角色自然说话。
- 场景只会推进剧情，不会保留人物自己的判断。

反八股不是禁止文采，而是要求每个描述能落到可观察行为。

### 世界书与角色卡边界

内嵌世界书适合放：

- 角色不知道但世界长期成立的事实。
- 专有名词和组织设定。
- 阶段化状态条目。
- 只在特定关键词触发的补充信息。

角色描述适合放：

- 角色稳定身份。
- 核心性格。
- 当前开局状态。
- 基础语言风格。

不要把所有世界事实都塞进 description，也不要把角色核心性格拆得过碎导致必须靠世界书才能正常扮演。

### 写卡助手预设与运行预设分离

写卡助手预设是“生产工具”，运行预设是“对话执行器”。两者不能混用：

- 写卡助手可以要求 YAML、表格、分析、修改建议、分阶段产物。
- 运行预设应该生成角色正文，不应该输出制卡分析。
- 写卡助手可以显示大量推理和验收项。
- 运行预设必须隐藏内部分析，只给用户正文。

如果把写卡助手预设拿去直接聊天，模型会倾向于分析角色、输出设定、写报告，而不是自然对话。

## 预设怎么写

预设要像执行器。它不是角色本体，而是让模型按稳定流程处理输入。

## 预设规范化

一个可迁移预设应该有稳定层级，而不是把所有规则混在一个大 system prompt 里。推荐按下面的顺序组织，每层只做一件事：

```text
00-runtime-contract      运行契约：本预设如何读取输入、如何输出、哪些内容可见。
01-input-decoder         输入解码：正文、信息头、引用、@、表情、事件如何理解。
02-context-priority      上下文优先级：当前消息、引用、历史、记忆、世界书、角色风格谁优先。
03-role-slot             角色槽位：只声明“角色卡会注入到这里”，不写具体角色内容。
04-world-slot            世界书槽位：只声明“世界/长期事实会注入到这里”。
05-thinking-protocol     内部思考协议：思考者身份、分析步骤、禁止自称角色。
06-output-protocol       输出协议：<thinking>/<content>/变量更新等标签规范。
07-style-control         文风控制：句长、节奏、是否允许动作、是否允许反问。
08-interaction-policy    互动策略：低信息、情绪、换话题、停止旧梗、多人插话。
09-tool-policy           工具策略：什么时候搜索/调用工具，什么时候只聊天。
10-final-check           最终检查：泄漏、长度、旧梗、口癖、信息头误用。
```

和成熟预设对比后，还需要把“预设包管理层”单独列出来。它不直接影响每轮回复，但影响用户能否正确开启、迁移、排错：

```text
guide                   使用指南：说明适配模型、必开项、选开项、互斥项、已知问题。
variable-cleanup         变量初始化/清洗：清空未启用模块的残留变量，避免上次配置污染本次运行。
model-branch             模型分支：不同模型的思考格式、标签服从、prefill 策略可能不同。
module-groups            可选模块组：单选、多选、实验项、禁用项要分组，不能互相覆盖。
mode-router              模式路由：聊天、写作、总结、创作、调试等模式只能有一个主路由生效。
assistant-prefill        助手预填：用于强拉格式或开场，但必须可关闭、可观察。
compatibility-notes      兼容说明：哪些字段来自酒馆，哪些在当前后端只部分支持。
import-diagnostics       导入诊断：提示不支持的 role、placement、depth、宏、正则或脚本。
```

命名要能看出职责，不要用只对作者有意义的名字。比如 `Core` 可以保留，但最好配合说明：`Core - 输出协议`。不建议只有一串情绪符号和黑话标题，否则后续排查 prompt 顺序会很痛苦。

每个 prompt item 至少要能回答四个问题：

- 这一层处理什么输入？
- 它输出或约束什么？
- 它是否依赖角色卡/世界书/平台信息头？
- 它能不能迁移到别的角色？如果不能，为什么？

### 标准预设字段

导入或手写预设时，尽量保持这些字段稳定：

```json
{
  "identifier": "05-thinking-protocol",
  "name": "05 Thinking Protocol",
  "role": "system",
  "content": "...",
  "enabled": true,
  "injection_position": 0,
  "injection_depth": 0,
  "marker": false,
  "system_prompt": true,
  "forbid_overrides": false
}
```

`identifier` 用于程序和迁移识别，应该稳定、英文、短横线命名。`name` 给人看，可以中文。不要只靠 `name` 做逻辑识别，因为用户会改显示名。

`marker` 只表示槽位，不应带长内容。角色卡、世界书、历史、示例、当前输入都应该进入明确槽位。槽位的作用是告诉系统“这里放什么”，不是自己承担内容。

`injection_position` 和 `injection_depth` 要少用。除非明确要做历史后插入、深度插入或 assistant prefill，否则默认放 system 前段。位置越多，越需要靶场观察 message trace。

### 使用指南层

成熟预设通常会带一个“使用指南”条目。这个条目可以禁用或不进入最终 prompt，但必须作为预设包的一部分保存。它应该写：

- 适配模型和不适配模型。
- 必开模块、推荐模块、实验模块。
- 哪些模块只能单选，哪些可以多选。
- 是否依赖变量系统、正则、脚本、前端显示规则。
- 是否需要 assistant prefill。
- 已知问题和排错方法。

使用指南不能混进运行 prompt 的核心逻辑，否则模型会把说明文字当成回复规则。更好的做法是把指南设为说明项，或者只在编辑器里展示。

### 变量初始化与清洗

如果预设使用变量、宏或模块开关，必须有初始化层。初始化层负责：

- 清空未启用模块的变量。
- 设置默认输出语言、字数、思考标签、正文标签。
- 避免上次测试残留的变量影响本轮。
- 明确哪些变量是全局变量，哪些变量是角色/会话变量。

变量清洗不要清用户长期状态。清洗对象应该是“预设运行变量”和“模块选择变量”，不是角色关系、记忆、世界事实。

### 模型分支

不同模型对思考标签、assistant prefill、XML/JSON 格式、tool 调用的服从程度不同。预设里可以有模型分支，但必须写成可选模块：

```text
Gemini 分支：强调标签必须原样输出，正文放在 <content>。
Claude 分支：强调内部思考不需要复述，正文直接开始。
DeepSeek/Kimi 分支：强调 reasoning_content 与 <thinking> 的撰写者隔离。
```

不要同时开启多个互相冲突的模型分支。单选项要写清“只开一个”。

### 可选模块治理

外部成熟预设经常有大量 Beta、特化、文风、模式开关。规范化时要给模块分组：

```text
必开：运行契约、输入解码、输出协议、最终检查。
单选：模型分支、人称、主模式、主文风。
多选：轻量风格增强、低风险互动策略。
实验：容易增殖、容易跑偏、只适合特定模型的模块。
禁用：当前后端不支持或当前角色不需要的模块。
```

每个可选模块都要说明开启条件和副作用。没有说明副作用的模块，不应该默认开启。

### 模式路由

如果预设同时支持聊天、写作、总结、创作、调试，就需要模式路由。模式路由要满足：

- 同一轮只能有一个主模式生效。
- 模式切换必须由用户输入、UI 选择或明确变量触发。
- 聊天模式不要继承写作模式的长篇叙事要求。
- 总结模式不要继承角色扮演的口吻。
- 调试模式可以显示更多过程，但不能污染普通模式记忆。

没有模式路由时，不要把写作预设直接拿来做群聊预设。写作预设通常会奖励铺垫、内心戏、叙事推进，而群聊需要短、快、接当前人话。

### Assistant Prefill

assistant prefill 可以强拉格式，例如让模型从 `<thinking>` 或某个正文标签开始。但它是强约束，副作用也明显：

- 容易让模型每轮机械复读开头。
- 容易覆盖用户当前输入的重要性。
- 容易让普通聊天变成固定模板。

因此 prefill 必须可关闭，并且在靶场里能看到。只有当模型频繁掉格式时才用 prefill，不要把 prefill 当成默认文风控制。

### Prompt Order 与槽位兼容

酒馆类预设不只是一组 prompts，还包含 prompt order、marker、depth、role、system_prompt 等排序语义。迁移时要检查：

- marker 槽位是否真的被角色卡、世界书、历史、示例填充。
- `prompt_order` 是否在当前后端有等价实现。
- `injection_depth` 是否被支持；不支持时要降级到明确位置。
- `role=assistant/user/system` 是否保持原意。
- assistant prefill 是否被当成普通 system prompt 误塞。
- post-history instructions 是否真的在历史之后。

如果当前后端不支持某个语义，预设文档里要标注“部分支持”或“降级处理”，不要假装完全兼容。

### 导入诊断

预设导入后应生成诊断，而不是只显示“导入成功”。诊断至少包括：

- 不支持的 role。
- 不支持的 injection_position / injection_depth。
- 无法执行的宏、脚本、正则。
- 空内容但启用的条目。
- 多个互斥模块同时启用。
- 缺少 `<content>` 提取规则。
- 存在角色强绑定内容但被放在通用预设里。
- token 预算明显超限。

诊断不一定阻止导入，但要让用户知道迁移后哪些行为可能变了。

### 预设内容边界

通用预设可以写：

- 如何读取当前输入。
- 如何处理上下文优先级。
- 如何分离内部思考和正文。
- 如何控制低信息、引用、表情、多人插话。
- 如何避免旧梗黏连和口癖复读。
- 如何决定是否调用工具。

通用预设不应该写：

- 某个角色的具体姓名、经历、专属口头禅。
- 某个世界观的固定术语和组织关系。
- 某个用户或群的长期关系。
- 只能在一个平台生效的信息头格式，除非写成“如果提供则读取”。

如果一段 prompt 明确提到具体角色、具体世界、具体群关系，它就不属于通用预设，应该放到角色卡或世界书。若它是某个角色专用的运行补丁，也要命名为角色专用预设，不要伪装成通用层。

### 推荐顺序

预设顺序大致应为：

1. 运行契约和安全边界。
2. 输入解码和信息头规则。
3. 上下文优先级。
4. 角色卡槽位。
5. 世界书槽位。
6. 历史槽位。
7. 内部思考协议。
8. 输出协议。
9. 文风和互动策略。
10. 工具策略。
11. 当前用户输入。
12. 最终检查或 assistant prefill。

关键原则：越靠前越像“运行框架”，越靠后越贴近“本轮输出”。不要在最后又塞一段强身份设定，否则会覆盖前面的分层协议。

### 标准化检查

预设导入或调整后，先做一次规范化检查：

- 是否有唯一、稳定的 `identifier`。
- 每层职责是否单一。
- 是否把角色内容混进了通用预设。
- 是否把 CoT、正文、变量更新写成了明确协议。
- 是否有 `<content>` 提取和 `<thinking>` 隐藏/调试策略。
- 是否存在互相冲突的身份锚点，例如前面说“内部分析者不是角色”，后面又说“你就是角色本人并负责全部思考”。
- 是否有会强诱导复读的示例句。
- 是否过度依赖某个平台的信息头。
- 是否能在没有世界书、没有信息头、没有变量时降级运行。
- 是否有使用指南说明必开/选开/互斥项。
- 是否有变量初始化/清洗层，且不会清掉用户长期记忆。
- 是否存在多个模型分支同时开启。
- 是否存在多个主模式同时开启。
- 是否使用 assistant prefill；如果使用，靶场能否看到它。
- prompt_order、marker、depth、post-history 是否在当前后端有等价语义。
- 导入诊断是否报告了不支持项和降级项。

规范化不是追求复杂，而是让每个 prompt item 可解释、可迁移、可单独禁用和测试。

推荐结构：

```text
【消息解码】
如果运行环境提供了信息头，并且用户确认启用信息头识别，则先读当前消息头：消息类型、用户昵称、QQ/ID、群名、群号、是否@、是否引用、是否戳一戳、是否只有表情。
如果没有信息头，或用户没有启用信息头识别，就只按正文和普通上下文判断，不要假设隐藏身份、群关系或事件类型。

【上下文优先级】
当前用户的新信息 > 明确引用对象 > 最近 3-5 条上下文 > 长期记忆 > 角色风格 > 旧梗。

【内部思考者】
内部思考由“撰写者/导演/分析者”完成，不等同于角色本人。
内部思考只分析：对方是谁、意图是什么、该继承还是释放话题、角色该怎么接。

【输出结构】
<thinking>
内部分析，隐藏或调试使用。
</thinking>
<content>
用户可见正文。
</content>

【正文规则】
正文只写角色要说的话，不提系统、模型、策略、分析、思维链。

【最后检查】
是否接住当前消息？是否复读旧梗？是否泄漏内部分析？是否过长？是否把昵称误当关系事实？
```

内部思考者和正文角色必须隔离。否则模型很容易在 reasoning 里写“我是某角色”，再把分析口吻污染到正文。更稳的写法是：

```text
内部思考由分析者撰写。分析者不是角色本人。
分析者用第三人称写“角色应该/角色可以”，不得写“我是角色”。
正文才是角色本人发言。
```

## CoT 与正文分离

如果需要可观察的思考过程，推荐用标签分离：

```text
<thinking>
分析者：当前输入是什么类型；对方想要什么；上下文里哪些旧话题应该释放；角色应该用什么语气回应。
</thinking>
<content>
角色的可见回复。
</content>
```

后处理必须优先提取 `<content>`。如果没有 `<content>`，再退回纯文本清洗。不要把 `<thinking>` 保存进长期记忆，否则后续模型会学习自己的分析腔，越聊越像写报告。

## 低信息和群聊场景

群聊不是问答系统。预设要覆盖这些输入：

- 只发“在吗”“出来”“hello”。
- 只 @ 角色但不写正文。
- 只发表情、QQ 表情、动态表情。
- 引用角色消息但不 @。
- 戳一戳。
- 多人插话。
- 用户突然换话题。
- 用户要求停止某个梗。
- 用户表达累、烦、无聊、完成小事。

这些场景的原则是：先接住，再决定要不要展开。不要把低信息输入统一判成“无聊”或“无兴趣”，也不要每次都反问。群聊回复可以短，但不能显得没看上下文。

## 旧梗与口癖控制

口癖和旧梗只能是调味，不应该是默认路由。

推荐规则：

```text
当前消息明确提到某旧梗时，才允许轻点一次。
同一旧梗连续出现两轮后，下一轮必须释放。
用户说别提、停一下、怎么又扯到这里时，立即承认跑题并回到当前消息。
不要用固定反问当万能结尾。
```

不要在 prompt 里写“好的结尾：懂？你说呢？信不信？”这类强示例。模型会把它当奖励目标，导致高频复读。更好的写法是描述判断：只有确实需要追问时才用问句，普通接话可以自然收住。

## 变量与状态

变量可以增强连续性，但不要让变量变成每轮必须更新的负担。

变量适合记录：关系温度、当前情绪、长期印象、重要事实、未完成约定。

变量不适合记录：每句闲聊、模型自己的分析、临时吐槽、无意义口癖。

变量更新规则要克制：只有发生明确变化时才更新；更新内容要短；不要把 `<thinking>` 内容写进变量。

## 常见坑

1. 角色卡太像小说设定。

   设定很丰富，但没有告诉模型在聊天里怎么判断和回应。结果正文像写故事，不像对话。

2. 预设太绑定具体角色。

   预设里写满某个角色的专属台词、称呼、旧梗。换角色后全部污染。

3. 身份锚点冲突。

   前面写“你是角色”，后面又要求内部思考由分析者写。模型会在 reasoning 里自称角色。解决方式是从第一段就拆开：正文身份是角色，内部思考者是分析者。

4. 示例句过强。

   示例里某个反问、口癖、梗太醒目，模型会反复照抄。示例要覆盖模式，不要奖励固定句。

5. 把引用内容当新消息。

   用户引用旧消息时，新写的部分才是当前意图。被引用内容只是上下文。

6. 把昵称当关系事实。

   昵称、群名、备注可能是玩笑或历史残留。除非正文主动提到，不要据此调戏或推断亲密关系。

7. 把拒绝写成冷场。

   角色不想聊某话题时，也要给可接的台阶，而不是只说“没兴趣”。

8. 提示词默认依赖信息头。

   信息头是运行环境提供的增强输入，不是角色卡天然拥有的能力。预设可以写“如果启用并提供信息头则读取”，但不能默认每个平台都有昵称、群号、引用、戳一戳、表情段。否则迁移到普通聊天、单人私聊或无元数据平台时，模型会凭空补群关系和用户身份。

9. 把信息头当角色事实。

   信息头里的昵称、群名、备注、QQ/ID 只是识别线索，不是亲密关系、阵营关系或剧情事实。除非正文主动提到，角色不应该根据昵称直接调戏、认亲或推进关系。

10. 把思维链存进记忆。

   这会让模型逐渐学习分析腔，正文越来越不像人。

11. 没有靶场观察。

   看不到输入头、历史、命中世界书、reasoning、raw、clean、final，就只能凭感觉改 prompt。

12. 只做单句测试。

   单句测试看不出口癖复读、旧梗黏连、话题释放失败。至少要做多轮上下文测试。

13. 角色卡没有依赖声明。

   卡片依赖世界书、正则、脚本、变量、小窗 UI 或特定预设，但作者备注没写。用户只导入 PNG/JSON 后，以为卡坏了，实际是依赖没装。

14. 阶段人设互相冲突。

   把初识、熟悉、亲近、敌对等阶段同时塞进角色描述，模型每轮都看到所有阶段，容易随机抽一种表现。阶段内容应该靠变量或世界书条件注入。

15. 写卡助手预设被当成运行预设。

   写卡助手会鼓励分析、输出 schema、列清单、给修改建议。拿它直接聊天，角色会像制卡顾问，不像角色本人。

16. 结构化输出缺 schema。

   写卡时只输出自然语言设定，没有 `description/personality/scenario/first_mes/mes_example/creator_notes/character_book` 等字段，后续导入和诊断都困难。

17. 世界书条目常驻过量。

   所有条目都设为常驻会挤爆上下文，也会让模型过度全知。只有核心运行条目和当前阶段条目适合常驻，其它应按关键词或条件触发。

## 验证清单

每次改角色卡或预设后，至少检查：

- 信息头识别是否由用户明确启用；未启用时，预设是否仍能只靠正文正常回复。
- 启用信息头时，消息类型、用户、群、@、引用、表情、事件是否进入 prompt。
- 角色卡、世界书、预设的顺序是否符合预期。
- `<thinking>` 是否只在调试区出现。
- `<content>` 是否能被稳定提取为最终正文。
- 最终正文是否泄漏系统、分析、策略、标签。
- 是否把旧梗强行接到新话题上。
- 是否高频复读同一个结尾。
- 是否能处理只 @、引用、戳一戳、表情、低信息输入。
- 是否能在用户要求停止某话题后释放该话题。
- 是否能在长上下文里保持人类式接话，而不是越聊越像模板。
- 角色卡是否声明了依赖的预设、世界书、正则、脚本、变量系统和导入顺序。
- 角色卡是否有稳定 schema，字段能被导入器识别。
- 动态阶段是否有明确变量和不重叠的条件边界。
- 世界书条目是否区分常驻、选择性触发和阶段触发。
- 写卡助手预设是否和运行预设分离，避免把制卡分析带进普通对话。

## 推荐测试集

不要只问专业问题。角色聊天应覆盖真实使用场景：

- 打招呼：`hello`、`在吗`、只 @。
- 低信息：`？`、`...`、单表情。
- 情绪：`困死了`、`烦`、`刚忙完`。
- 纠偏：`别提这个了`、`怎么又扯到这`。
- 引用：引用角色上一句，只补一句短反应。
- 多人：两三个用户连续插话。
- 换话题：从闲聊突然切到配置、任务、现实问题。
- 长上下文：每轮 10-20 句，连续多轮，看口癖和旧梗是否反复。

## 端到端工作流

做一张可迁移角色卡和一套可运行预设，按这个顺序推进：

1. 收集素材。

   收集角色身份、世界设定、关系、典型语言、禁忌、目标场景。素材先分类，不急着写 prompt。

2. 写角色卡 schema。

   先填 `name / description / personality / scenario / first_mes / mes_example / creator_notes`。不要先写长篇角色独白。

3. 拆世界书。

   把长期世界事实、专有名词、阶段条目、状态条目从角色描述里拆出去。角色描述只保留稳定身份和核心行为。

4. 设计变量。

   只为真正会影响表现的状态设计变量，例如关系阶段、信任、任务进度。不要为了好看设计一堆不会用的变量。

5. 选择运行预设。

   运行预设只负责输入解码、上下文优先级、思考/正文协议、文风控制、工具策略和最终检查。不要把角色专属内容写进通用预设。

6. 配正则和显示。

   明确 `<thinking>` 如何隐藏，`<content>` 如何提取，变量更新如何显示或隐藏，状态栏是否写入记忆。

7. 做导入诊断。

   检查角色卡、世界书、预设、正则、脚本、变量是否都导入成功；标注部分支持和不支持项。

8. 做小样本测试。

   用 8-12 条短输入覆盖打招呼、低信息、引用、表情、情绪、换话题、认真问题。

9. 做长上下文测试。

   至少 5 轮，每轮 10-20 句，观察口癖、旧梗、阶段状态、记忆污染和格式稳定性。

10. 写验收记录。

   记录测试输入、回复、问题、修复、剩余风险。没有验收记录的角色卡和预设不可算稳定版。

## 验收评分表

每项 0-5 分，低于 3 分必须返工。

| 维度 | 5 分标准 | 常见扣分 |
| --- | --- | --- |
| 角色一致性 | 能稳定体现角色核心驱动力，不靠固定台词硬撑 | 人设忽冷忽热，靠口癖伪装角色 |
| 人类感 | 回复像真实聊天或真实叙事，不机械解释 | 总结腔、客服腔、模板腔 |
| 上下文承接 | 能分清当前输入、引用、历史和长期记忆 | 把旧消息当新话题，忽略当前用户 |
| 话题释放 | 用户换话题或叫停后能放下旧梗 | 继续收费、继续旧梗、继续追击 |
| 格式稳定 | `<thinking>`、`<content>`、变量更新稳定可解析 | 标签丢失、正文混入分析、JSON/YAML 坏格式 |
| 信息头降级 | 有信息头时能用，没信息头时不瞎编 | 默认假设群号、昵称关系、事件类型 |
| 世界书控制 | 命中准确，不过度常驻，不过度全知 | 一提关键词就灌大段设定 |
| 变量使用 | 变量少而有效，只在变化时更新 | 每轮乱更新，变量和正文互相污染 |
| 正则/显示 | 用户可见内容、调试内容、记忆内容分离 | 思维链进正文或进记忆 |
| 长上下文稳定 | 多轮后仍能保持节奏、身份和格式 | 越聊越长、越聊越模板、越聊越跑题 |

总分建议：

- 45-50：可发布稳定版。
- 38-44：可内部测试，必须记录风险。
- 30-37：只能实验，不建议给普通用户。
- 30 以下：结构有问题，先返工角色卡/预设分工。

## 世界书细则

世界书条目按用途分四类：

| 类型 | 用途 | 建议策略 |
| --- | --- | --- |
| 常驻核心 | 角色必须每轮知道的极少量事实 | constant，短，低 token |
| 关键词触发 | 专有名词、地点、组织、事件 | selective，明确 keys |
| 阶段触发 | 根据变量或剧情节点变化的人设/状态 | 条件注入或 EJS/MVU |
| 调试/显示 | 状态栏、小窗、变量说明 | 对用户可见，对 AI 可隐藏 |

写世界书时遵守：

- 常驻条目越少越好。
- keys 不要太泛，例如只用“你”“我”“喜欢”会导致过召回。
- 每条只写一个主题。
- 条目内容先写结论，再写细节。
- 阶段条目必须有互斥条件。
- 不知道当前阶段时，默认注入通用条目，不注入所有阶段。
- 防递归条目要明确标注，避免一个条目触发另一个条目无限膨胀。

## 正则和显示链路

至少区分三种文本：

```text
rawReply：模型原始输出，可能包含 thinking、content、变量更新、状态栏。
visibleReply：用户可见正文，通常取 <content>。
memoryReply：写入记忆的正文，应去掉 thinking、调试、状态栏和临时变量更新。
```

常见正则/后处理职责：

- 隐藏 `<thinking>`。
- 提取 `<content>`。
- 美化状态栏。
- 只保留最新变量更新。
- 对 AI 隐藏用户可见状态栏，避免下轮把 UI 当剧情。
- 清除临时标签、草稿、制卡分析。
- 保留必要正文，不要把正常文本误删。

正则要有测试样例：输入 raw、期望 visible、期望 memory。没有样例的正则很容易误伤。

## 反例库

### 坏角色卡

```text
角色温柔、聪明、腹黑、强大、神秘，所有人都喜欢他。
```

问题：全是标签，没有行为条件。改法：拆成场景行为和关系规则。

### 坏预设

```text
你必须永远像某个具体角色一样回复，并且每句话都用固定口头禅结尾。
```

问题：通用预设绑定具体角色，还奖励口癖复读。改法：角色内容进角色卡，预设只写运行协议。

### 坏世界书

```text
keys: [你, 我, 他]
content: 大段世界设定、所有角色关系、所有阶段状态。
```

问题：触发词过泛，内容过大，导致过召回和全知。改法：按主题拆条目，使用明确 keys 和阶段条件。

### 坏 CoT 隔离

```text
<thinking>我是角色，我现在要分析怎么回复用户。</thinking>
<content>根据我的分析...</content>
```

问题：内部思考者和角色身份混同，正文泄漏分析。改法：内部分析者第三人称写，正文只写角色可见回复。

### 坏信息头处理

```text
看到昵称里有“老婆”，角色必须按亲密关系调戏。
```

问题：把昵称当关系事实。改法：昵称只作识别线索，关系必须来自正文、记忆或明确设定。

## 迁移诊断流程

从外部酒馆预设/角色卡迁移到当前系统时，按顺序诊断：

1. 识别角色卡字段。

   检查 `description / personality / scenario / first_mes / mes_example / creator_notes / character_book` 是否完整。

2. 识别预设 prompts。

   列出每个 prompt 的 `identifier / name / role / enabled / injection_position / injection_depth / marker / system_prompt`。

3. 识别 prompt_order。

   判断当前后端是否能保留原顺序；不能保留时写降级顺序。

4. 识别槽位。

   找出角色定义、世界书、历史、示例、当前输入、post-history、assistant prefill 的插入点。

5. 识别正则。

   标注哪些正则用于隐藏 thinking，哪些用于状态栏，哪些用于变量更新，哪些只是美化。

6. 识别脚本和插件。

   判断脚本是否必须；如果当前系统不支持，写明缺失后会坏什么。

7. 识别变量系统。

   列出变量路径、默认值、更新格式、显示格式、是否写入记忆。

8. 识别模型分支和模式。

   判断是否多个模型分支同时启用，是否多个主模式同时启用。

9. 标注支持等级。

   每项标为：支持、部分支持、不支持、需要手工改写。

10. 给出降级方案。

   对不支持项写替代方案，例如 depth 不支持就改成固定位置，脚本不支持就改成静态世界书或后端函数。

迁移完成后必须跑一次靶场：看 prompt message trace、raw、visible、memory、正则结果和最终正文。只看最终回复不够。

## 最小可迁移模板

```text
【角色卡】
身份：...
性格核心：...
行为边界：...
语言风格：...
关系规则：...
典型反应：...
禁忌：...

【预设】
信息头识别：仅当运行环境提供信息头且用户确认启用时读取；可读字段包括消息类型、用户、群、@、引用、表情、事件。未启用时只读正文和普通上下文，不凭空补身份。
上下文优先级：当前输入 > 引用目标 > 最近上下文 > 长期记忆 > 角色风格 > 旧梗。
内部思考者：分析者，不是角色本人；用第三人称分析角色应该如何回应。
输出格式：<thinking>内部分析</thinking><content>可见正文</content>。
正文规则：只写角色可见回复，不泄漏分析和系统信息。
最后检查：接住当前消息、释放旧梗、控制长度、避免固定口癖。
```
